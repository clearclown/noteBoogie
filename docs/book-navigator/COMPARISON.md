# NotebookLM / Open Notebook / noteBoogie 比較

フォーク元 Open Notebook の「vs Google Notebook LM」表を3者比較に拡張したもの。
分析の全文は §3〜§5、設計思想の対比と「次の一手」は [ADVANCED_ROADMAP.md §4](ADVANCED_ROADMAP.md) を参照（2026-07-23 壁打ちの整理）。

## 1. 機能比較表

| 観点 | Google NotebookLM | Open Notebook | noteBoogie（本フォーク） |
|---|---|---|---|
| **プライバシー/データ主権** | Google クラウドのみ | セルフホスト、データは手元 | 同左 + **完全ローカル生成構成**（Ollama + OpenAI互換ローカルTTS）まで可 |
| **AIモデル選択** | Google モデル固定 | 18+ プロバイダ | 18+ プロバイダ + **生成ごとにGUI/CLIで選択**、品質×コストの**実測比較ハーネス**付き |
| **日本語縦書きの紙書籍** | スキャンPDFの構造推定なし（OCR品質任せ） | 汎用取り込みのみ（縦書き・柱・表の概念なし） | **YomiToku OCR + 右→左段組 + 章検出 + 表のMarkdown復元 + 図切り出し**を決定的パイプラインで実行 |
| **音声生成** | 音声概要（2話者・検証なし） | ポッドキャスト（1〜4話者・プロファイル自由） | **章別メンター独話オーディオブック** + 連続再生 + 図ギャラリー + 章単位リトライ |
| **生成物の検証** | なし（捏造しても観測不能） | なし | **自動採点4指標**（構成遵守/捏造リスク/敬体/長さ）+ LLM judge、スライドは**品質ゲート**（閾値未達を明示） |
| **プロンプト品質管理** | 不可（ブラックボックス） | 手動編集 | **RL最適化ハーネス**（報酬 = 品質 − λ·トークン、実測 0.688→0.732） |
| **ハルシネーション対策** | プロンプト頼み・構造的に検証層なし | 対策層なし（透明性はある） | **3層防御**: 入口の決定論化 + 章分割ガードレール + 出力の独立採点（§3） |
| **検索/RAG** | 静的インデックス top-k（+ long context 詰め込み） | 全文 + ベクトル検索 | ベクトル検索 + **学習傾斜**（本/章 0〜2 × 自動傾斜）再ランク + メンター長期記憶 |
| **引用** | 包括的（ただし完全ではない） | 基本レベル（改善予定） | 本文グラウンディング ask（`[source:…]` 引用）+ 図キャプション出典 |
| **コーチ機能** | なし | なし | **メンターAI**（蔵書+記憶+傾斜+音声回答）、**スライドレビュー5軸**、**pptx 非破壊自動修正**（`_coached.pptx`） |
| **失敗の可観測性** | ブラックボックス | OSS + ログ | `generation_error` 可視化 + **テスト 2,500+ / CI 8ジョブ** + 密閉E2E + 実ブラウザ検証 |
| **API / 外部連携** | なし | REST API | REST + Rust gateway + gRPC sidecar + **MCP**（Claude Desktop/Code から蔵書を検索・質問・相談） |
| **デプロイ** | Google ホストのみ | Docker / cloud / local | Docker/Podman 一発起動 + **Tailscale リモートアクセス** |
| **コスト** | 無料枠 + サブスク | AI 使用分のみ | AI 使用分のみ。実測 ~$10.9/冊（408頁）→ 節約構成 ~$3、ローカルTTSで **$0** |

## 2. ハルシネーション混入点の層別比較

「ハルシネーションはパイプラインの5箇所すべてから混入する」という整理に基づく（分析全文は §3〜§4）。

| 層 | NotebookLM（古典的RAG型） | Open Notebook | noteBoogie |
|---|---|---|---|
| **取り込み** | OCRノイズ・柱・目次がそのまま索引化 | 汎用パーサ（構造は推定しない） | ✅ **決定論化**: Rust+YomiToku で章・表・図・読み順を確定。LLMに構造推定させない |
| **チャンク化** | 固定長分割で文脈切断 | トークンベース分割 | 章・表・図マーカーを保った本文で埋め込み（表はMarkdownとして文脈に残る） |
| **検索** | 単発 top-k、外れたら一般知識で補完 | 全文+ベクトル | ✅ ベクトル + 傾斜再ランク + **Self-RAG 分岐**: 類似度下限（0.4、env較正可）未満は LLM を呼ばず「根拠なし」と決定的に断る（ask/メンター両方、判定は quality_event に記録） |
| **生成** | 根拠不足でも生成続行 | 同左 | ✅ **ガードレール**: H1のみ分割・同名章マージ・200字未満折り込みで「薄い素材からの捏造」を入口で遮断 |
| **二次生成物**（音声・図解） | 検証層なし（音声概要が勝手に補完する事例） | 検証層なし | ✅ **採点がゲート化済み**: TTS 前に自動採点→閾値0.6未満は批評付き再生成1回→なお未達なら棄却（費用をかけず失敗を可視化）。スライドも品質ゲート。**残る穴**: 任意の judge 指標は LLM で確率的（ゲート本体は正規表現ベースで決定的） |

## 3. 結論（設計思想の対比）

- **NotebookLM のハルシネーションは「たまたまのバグ」ではなく構成上の必然。** 実体は静的インデックスの古典的RAG + Gemini であり、①検索失敗時に一般知識で隙間が埋まる、②音声概要など生成的変換に検証層がない、③ブラックボックスで検証・修正が不可能、という3点が構造的弱点。
- **基礎技術の再考は「retrieval の高度化」だけでは足りない。** RAG→Agentic RAG は検索と生成の間に判断・検証ループを入れる点で有効だが、入口（OCR・構造化品質）と出口（生成物検証）の穴は塞げない。
- **Open Notebook は「ブラックボックス問題」への回答**（自己ホスト・プロバイダ選択・REST API・OSS）。ただしハルシネーション対策の3層はまだ持たない。
- **noteBoogie は3層防御の実装**: (a) 決定的前処理（Rust+OCR構造化）、(b) ルールベースガードレール（章分割）、(c) 自動採点 + RLプロンプト最適化。検索アルゴリズムの魔改造ではなく、**LLMが捏造できる余地そのものを削る品質工学**。

一言でまとめると: NotebookLM は「優秀なLLMを信頼する」設計、noteBoogie は「**LLMを信頼しない**」設計。LLMに任せるのは台本の文体と対話だけに絞り、構造化は決定的処理に、危険な入力はルールで排除し、出力は独立採点する。

## 4. 正直な限界（自己申告）

- 任意指定の judge 指標（LLM採点）は確率的（既定のゲート指標は正規表現ベースで決定的）
- Self-RAG の下限値 0.4 は経験則。実トラフィックの `quality_event` ログでの較正が今後の課題
- 全量変換の manifest 章リストに柱の再検出ノイズが残る（図の章対応精度に影響）
- chat は検索を持たない設計（コンテキストはユーザー選択）のため Self-RAG 分岐の対象外
