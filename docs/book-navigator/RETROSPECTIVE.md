# 開発振り返り（2026-07 時点）

Book Navigator を仕様策定から全量E2Eまで開発した過程の振り返り。事故・発見・学びを次の開発に引き継ぐための記録。

## 何を作ったか（結果）

- スキャン縦書き和書PDF → 章別メンター独話オーディオブック + 質問可能ナレッジベース + MCP外部公開
- Rust gateway（reinhardt-web）+ Python sidecar（gRPC）+ SurrealDB + Next.js の4層に、変換用の姉妹Rustリポを接続
- 品質装置: 台本自動採点ハーネス + RLプロンプト最適化（実測 reward 0.688→0.732、最適化器が「捏造禁止ルール」を自律追加）
- テスト: pytest 713 / vitest 179 / Playwright 3 / gateway 38 / converter 1546、CI 8ジョブ

## 重大インシデントと学び

### 1. 123章事件（コスト事故を実行中に捕捉）
全量生成の初回、章分割が **123章** を返した。縦書き実書籍は「第1部」「第1章」の柱がページ毎に**交互**出現し、連続同名マージでは吸収できなかった。放置すればTTSだけで$50超。
**対応**: 即時停止 → 「タイトルの実質的初出のみが章開始」ルール → 表記ゆれ正規化キー（123→29→13章）。
**学び**: スモーク（30頁）は前付けだけで本文パターンを含まなかった。**サンプルは文書の「代表的な構造」を含む区間で取る**。生成系のコスト操作は必ず件数を先に確認してから走らせる。

### 2. ほぼ空の章からの捏造（ハルシネーションの定量化）
30字の「章」からLLMが尤もらしい章解説（存在しない図の説明まで）を生成していた。評価ハーネスで定量化すると、薄い章のグラウンディングは 0.13〜0.38。
**対応**: 200字未満の章を折り込むガードレール + RL最適化が briefing に捏造禁止を自律追加。
**学び**: 「もっともらしさ」は目視では見抜けない。**グラウンディング率のような自動指標を先に作る**と、対策の効果も測れる。

### 3. upstream 129コミットのマージとマイグレーション番号衝突
自作 migration 16/17 が upstream の 16〜23 と衝突 → 24/25 へ改番。さらに upstream の「speaker_config record link 化」「audio_file 相対パス化」が自作 seed / gateway 書き込みを静かに壊す変更だった。
**学び**: フォークでの番号資源（migration 番号）は**衝突前提**で設計する（番号ギャップ監視テストを常設した）。マージ時は「schema を書く側」のコードを必ず追跡する。

### 4. CI が実は守っていなかった
gateway の cargo テストは**一度もCIで実行されておらず**、sidecar テストも依存グループ未導入で silent skip。push して初めて protoc 欠如・ruff・mypy の3ジョブが落ちた。
**学び**: 「テストがある」と「CIが回している」は別。**新しいテスト種別を足したら、CIで落ちることを一度確認する**（fail-first）。

### 5. ローカルで通るE2E、CIで落ちるE2E → 実バグ発見
Playwright がローカルで通ったのは**手元の実APIが未モックのリクエストに偶然応答していた**から。密閉化（キャッチオール）すると、今度は認証ガードのレースで /podcasts への deep link が /notebooks に消える**実バグ**（upstream 由来）が露出した。
**学び**: E2E は**密閉（hermetic）にして初めて信頼できる**。モックの高速応答はレース条件の検出器になる。

### 6. 二重処理の監査（ユーザー指摘起点）
白紙ページへの vision 課金、YomiToku が読んだ図内テキストを Claude に再読させる二重読取、構成固定の独話に outline+transcript の2パス。
**学び**: パイプラインは動いた後に**「どこで同じ情報を二度作っているか」を棚卸しする**。上流(OCR)の出力を下流(LLM)が使えるよう配管するだけで消える無駄が多い。

### 7. 環境の暗黙依存
venv の置き場所規約（ai_bridge/ai_venv）、DATA_FOLDER のハードコード（./data）、SIDECAR_ADDRESS と SIDECAR_GRPC_ADDR の二重名、DEEPSEEK_API_KEY のキー名。どれも「動かない」ではなく**静かに劣化**する類いだった。
**学び**: 暗黙の規約はREADMEではなく**トラブルシュート表**に落とす（症状→原因の逆引き）。

## うまくいったこと

- **Rust主体+Python sidecar** の分割: LLM/TTS の Python 資産（podcast-creator/esperanto）を書き直さず、オーケストレーションだけ Rust 化。モック gRPC サーバーで生成ループまで統合テストできた
- **評価ハーネス → RL最適化**の順で作ったこと: 報酬関数が先にあったので最適化が即座に機能し、比較実測（haiku 同点/-68%）も同じ装置で取れた
- **細かい論理コミット + 都度push**: 事故時の巻き戻し・CI検知が速かった
- **監査エージェントの定期投入**: 仕様との突合で compose パス切れ・CORS 欠如など「動くまで気づかない」欠陥を実行前に検出
